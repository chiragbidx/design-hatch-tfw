## Database & Prisma Rules (CRITICAL ZONE)
- `schema.prisma` is READ-ONLY by default. Changes require explicit proposal and approval.
- `prisma/migrations` MUST NEVER be hand-edited or deleted. No manual SQL migrations.
- No out-of-band DB schema changes. All schema changes go through Prisma CLI after approval.
- Allowed schema changes (only after approval): adding fields/tables with non-breaking defaults; adding indices.
- Forbidden without approval: breaking changes (drops/renames), custom SQL, seeding that mutates prod-like data.
- Change process:
  1) Propose change in writing (what/why/impact).
  2) Add/update `schema.prisma` in a diff after approval.
  3) Run `npx prisma migrate dev` locally; commit generated migration files unchanged.
  4) Do NOT edit generated migration SQL.
  5) Run `npx prisma generate` post-change.
- Seeding: use `prisma/seed.ts`; ensure idempotence and non-production data only.

## API Routes & Server Actions Contract
- Location: `app/api/**/route.ts` and any Server Actions. Contracts (paths, params, response shapes) are IMMUTABLE unless approved.
- Auth enforcement is mandatory on all sensitive routes. No unauthenticated data writes.
- Validation REQUIRED for all inputs (body, query, params). Reject on failure.
- MAY: add new routes with explicit contract docs; extend handlers with backward-compatible fields.
- MUST NOT: change HTTP methods, relax auth, change response shape, or skip validation without approval.
- Server Actions must be server-only, validate inputs, and respect auth/session constraints.

- No privilege escalation, role inference, or silent fallback to elevated roles.

## Recommended Workflow
1) Read this README fully; verify applicable rules.
2) Ensure `env.example` exists and matches current env usage; sync `.env.local` locally.
3) Pull latest `main`; run `npm install` if deps changed.
4) Make minimal diff in permitted areas; keep changes scoped.
5) For server/API/DB changes, add validation and auth checks; update docs/contracts.
6) Run relevant checks: `npm run lint`, `npm run build`, `npm run seed` (if applicable), and Prisma commands when schema changes are approved.
7) Summarize changes, risks, and testing in PR/commit messages.
8) If blocked or unsure at any step: STOP AND ASK.
